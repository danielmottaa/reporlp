Integração oficial do repor

Guia técnico para conectar ERP, PDV ou sistema de mercado ao repor

Esta documentação foi feita para o programador do mercado conseguir integrar com o repor sem adivinhação: o que precisa antes de começar, como autenticar, quais endpoints chamar, quais payloads enviar, quais respostas esperar e como tratar erros comuns.

Base URL oficial https://api.repor.app
Autenticação externa x-api-key
Fluxo atual Sistema externo envia dados para o repor
Idempotência eventId obrigatório por evento

Como funciona hoje

O repor já está apto a receber produtos, packs e baixas do sistema externo

A integração atual é orientada a entrada de dados no repor. Isso significa que o ERP, PDV ou sistema do mercado envia dados para a API do repor, e o repor passa a refletir esses dados no app em tempo real.

O que já funciona

  • Receber catálogo de produtos por EAN.
  • Receber packs / lotes com quantidade, validade e setor.
  • Receber vendas para dar baixa automática no repor.
  • Manter operação manual mesmo com integração ativa.
  • Tratar reenvio de eventos duplicados com segurança.

O que ainda não faz

  • Enviar cadastro manual do repor para o sistema do mercado.
  • Empurrar packs criados manualmente para ERP ou PDV.
  • Criar setores automaticamente a partir da integração externa.

Passo a passo

Fluxo recomendado para integrar um mercado ao repor

1

Ative a integração no app

No app do repor, acesse Ajustes > Automação Externa e ligue a integração.

2

Obtenha a chave de integração

Use POST /markets/current/integration/rotate-key com token JWT de admin. Guarde a chave apenas no backend do integrador.

3

Valide a conexão

Faça um GET /integrations/status com x-api-key. Se retornar status: ok, a integração está pronta.

4

Sincronize o catálogo

Envie os produtos para POST /integrations/products/upsert. O repor usa o EAN como chave principal do catálogo por mercado.

5

Sincronize packs / lotes

Envie os packs para POST /integrations/batches/sync com externalId, validade, quantidade e sectorId do repor.

6

Envie as vendas para baixa automática

Envie as vendas para POST /integrations/sales. O repor baixa os packs por EAN, priorizando os mais próximos do vencimento.

Checklist

Pré-requisitos antes da primeira chamada

1. Mercado com integração ativada

Sem isso, a API externa não deve ser considerada pronta para operação.

2. Chave de integração gerada

O sistema externo precisa enviar x-api-key em todas as chamadas externas.

3. Setores já cadastrados no repor

Os packs exigem sectorId existente no repor. A integração não cria setor automaticamente.

4. Estratégia de eventId

Defina um identificador único por evento para permitir reprocessamento seguro sem duplicidade.

API

Endpoints relevantes para a integração

Endpoints administrativos de apoio

GET/markets/current/integration

Retorna a configuração atual da integração do mercado autenticado.

Auth: Authorization: Bearer <jwt-admin>

Uso prático: validar se o mercado está com a integração ligada e se já existe chave ativa.

PATCH/markets/current/integration

Ativa ou desativa a integração e ajusta preferências como intervalo de sync.

Body: { enabled, realtimeUpdates?, syncIntervalSeconds?, soldOutLabel?, providerName? }

POST/markets/current/integration/rotate-key

Gera uma nova chave de integração para o sistema externo.

Resposta: { apiKey, integration }

Importante: armazene a chave com segurança. Ela não deve ser exposta no frontend do sistema integrador.

GET/me/snapshot

Útil para obter dados internos do mercado, incluindo setores e respectivos IDs.

Observação: os sectorId usados na integração de packs precisam existir no repor.

Endpoints que o sistema do mercado deve consumir

GET/integrations/status

Healthcheck da integração. Use para validar se a chave está correta.

Header: x-api-key

curl -X GET https://api.repor.app/integrations/status   -H "x-api-key: SUA_CHAVE"
POST/integrations/products/upsert

Cria ou atualiza produtos do catálogo do repor por EAN.

Header: x-api-key

Limite: até 500 produtos por chamada.

Regra: se o mesmo EAN já existir no mercado, o catálogo é atualizado.

POST/integrations/batches/sync

Cria ou atualiza packs / lotes do repor.

Chave do lote: externalId

Header: x-api-key

Limite: até 500 packs por chamada.

POST/integrations/sales

Envia vendas para o repor dar baixa automática na quantidade dos packs por EAN.

Header: x-api-key

Limite: até 500 itens por chamada.

Regra: o repor baixa primeiro os packs elegíveis mais próximos do vencimento.

Respostas

O que esperar de volta da API

EndpointResposta de sucessoObservação
GET /integrations/status{ "status": "ok", "marketId": "...", "marketName": "..." }Use para validar a chave antes de enviar dados reais.
POST /integrations/products/upsert{ "status": "processed", "eventId": "...", "importedProducts": 10 }Se o mesmo eventId voltar, a resposta tende a ser duplicate.
POST /integrations/batches/sync{ "status": "processed", "eventId": "...", "syncedBatches": 25 }Atualiza lote existente por externalId ou cria novo.
POST /integrations/sales{ "status": "processed", "eventId": "...", "appliedItems": 3, "notes": [...] }Se nenhuma baixa for possível, pode responder ignored.

Erros comuns

Como interpretar e tratar os retornos mais importantes

401 Unauthorized

Ocorre quando a x-api-key é inválida, ausente ou desatualizada.

Como tratar: valide se a chave está correta, se o header está sendo enviado e se a chave não foi rotacionada no repor.

422 Unprocessable Entity

Ocorre quando o payload está inválido: EAN inválido, sectorId incorreto, data em formato errado, etc.

Como tratar: registre o corpo do erro, corrija o payload e reenvie com um novo eventId se for um novo evento.

202 with status: duplicate

O mesmo eventId já foi processado anteriormente.

Como tratar: considere a operação idempotente e não gere novo envio só porque retornou duplicate.

202 with status: ignored

Mais comum em vendas quando não havia packs elegíveis para baixa.

Como tratar: inspecione o campo notes e decida se precisa de ação operacional ou apenas logar o evento.

Regras importantes

Pontos que o integrador precisa saber antes de começar

1. O eventId é obrigatório

Cada envio precisa ter um eventId único. Se o mesmo evento for reenviado, o backend trata como duplicado e responde 202 com status: duplicate.

2. O EAN precisa ser válido

O repor valida EAN-8 e EAN-13. Se vier inválido, a API retorna erro de validação.

3. O sectorId precisa existir no repor

A integração de packs não cria setores automaticamente. O setor já precisa estar cadastrado.

4. Manual e integração coexistem

Mesmo com a integração ativa, o mercado continua podendo trabalhar manualmente dentro do app.

Cenários reais

O que acontece em situações comuns

Se o mercado cadastrar manualmente com a integração ativa, isso sobe para o ERP?

Não. Hoje a integração é de entrada no repor. O cadastro manual continua funcionando, mas não é enviado automaticamente para o sistema do mercado.

Se já houver produtos manuais e a integração for ativada depois?

Os dados manuais permanecem no app. Quando o sistema externo começar a sincronizar, produtos com o mesmo EAN podem complementar ou atualizar o catálogo do repor.

Se a venda chegar e não houver packs disponíveis para aquele EAN?

O repor registra a tentativa e devolve notas indicando ausência de packs elegíveis para baixa.

Se a chamada for repetida por timeout?

Reenvie com o mesmo eventId. A API é preparada para tratar duplicidade de forma segura.

Se o sistema externo quiser desligar a integração?

O desligamento é administrativo e deve ser feito pelo fluxo autenticado do mercado, via PATCH /markets/current/integration com token de admin.
